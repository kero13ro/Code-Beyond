非技術團隊要更新內容，這個需求會讓 CMS 選型的答案完全不同。

## 情境

三個品牌行銷網站，需求如下：
- 宣傳產品、聯絡資訊、職缺訊息
- 由行銷、美術、人資、翻譯部門不定期更新
- SEO 與效能是核心指標

關鍵限制：**主要編輯者是非技術人員**。這一點決定了所有選擇。

## 前端框架：Astro 而非 Next.js

靜態行銷網站不需要 React runtime 的開銷。Astro 預設輸出零 JavaScript，只在需要互動的組件（例如職缺表單）才載入 JS——這正是「島嶼架構」的優勢。

Next.js 更適合需要大量動態功能的應用；對以內容為主的行銷網站，它帶來的複雜度超過它解決的問題。

## Headless CMS 比較

| CMS | 編輯體驗 | 自訂彈性 | 託管 | 價格 |
|-----|---------|---------|------|------|
| **Storyblok** | 視覺化即時預覽 | 中 | SaaS | 較高 |
| **Contentful** | 結構化表單 | 高 | SaaS | 企業級 |
| **Strapi** | 結構化表單 | 完全自訂 | 自行託管 | 開源免費 |
| **Sanity** | 可自訂 Studio | 極高 | 混合 | 中等 |

**Storyblok**：視覺化編輯器是核心優勢——非技術編輯者能直接在預覽頁面上操作內容，不需要理解「content model」的概念。內建多語言、圖片 CDN、版本歷史。缺點是擴充後端邏輯的空間有限，價格在需求增加後會明顯上升。

**Contentful**：企業級穩定性，REST + GraphQL API 都支援。但編輯體驗偏向「填表單」，對非技術人員的學習曲線比 Storyblok 高。適合預算充足、內容結構複雜的大型專案。

**Strapi**：完全開源，自行託管代表完全控制——資料、API、schema 都可以任意修改。代價是需要技術團隊維護伺服器和資料庫，大流量時還要自己處理效能優化。適合技術團隊夠強、想控制長期成本的情境。

**Sanity**：開發者體驗最好——GROQ 查詢語言強大，Studio 可以用 React 完全自訂。問題是「自訂 Studio」本身需要 React 知識，讓非技術編輯者上手前，開發者要先投入設置時間。

## 決策

這個情境的選型邏輯：

- **非技術編輯者是主體** → 視覺化編輯器權重最高 → Storyblok
- **若技術團隊能支撐維運、且想控制成本** → Strapi
- **若需要兼顧開發彈性與編輯友善** → Sanity
- **若是大型企業、預算不是問題** → Contentful

最終選擇 **Storyblok**：非技術部門能自助更新是這個專案的核心需求，視覺化編輯器直接降低了溝通與培訓成本，這個價值比省下的 SaaS 費用更重要。

架構組合：Astro + Storyblok + Cloudflare Pages。靜態輸出、全球 CDN、內容由各部門自行維護，工程師不需要為每次文案修改待命。
